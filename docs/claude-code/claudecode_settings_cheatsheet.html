設定早見表.

Claude Code — settings / permissions / hooks / memory / MCP

設定は「チームで効かせる」ためにある。共有はコミット、個人は local、強制はフック。

どこに何を書くか・優先順位・許可制御・ゲート化までを1枚に。現行スキーマは変わり得るので /config・/status で確認を。

SHARED
.claude/settings.json
チーム共有の設定。コミットして全員に効かせる。permissions・hooks・env の本体。
PERSONAL
.claude/settings.local.json
自分用の上書き。gitignore 対象。一時的な許可や個人の好みはここへ。
CONTEXT
CLAUDE.md
毎回読まれるプロジェクト知識。規約・コマンド・“やってはいけない罠”を簡潔に。
01

ファイルの全体像

LAYOUT

「ユーザー単位（全プロジェクト）」と「プロジェクト単位（このリポ）」の2層。

# ユーザー単位（全プロジェクトに効く） ~/.claude/ ├── settings.json # 個人の既定（モデル・hooks・allow等） ├── CLAUDE.md # 個人のグローバル指示 ├── commands/ agents/ skills/ # 自分用の拡張 ~/.claude.json # ★別物：テーマ/セッション/MCP(user,local)。settings.jsonと混ぜない # プロジェクト単位（このリポだけ） <repo>/ ├── CLAUDE.md # 共有のプロジェクト知識（コミット） ├── .mcp.json # 共有MCPサーバ（コミット） └── .claude/ ├── settings.json # 共有設定（コミット） ├── settings.local.json # 個人上書き（gitignore） ├── commands/ agents/ skills/ └── hooks/ # フック用スクリプト
定番の罠：~/.claude.json は settings.json とは別ファイル。ここに settings.json のキーを入れるとスキーマエラーで黙って無視される。
02

設定の優先順位

PRECEDENCE

上が強い。上のスコープが下を上書きする。どれが効いてるかは /status の Setting sources で確認。

#スコープ場所用途
1Managed（組織）OS別の管理パス（managed-settings.json）会社方針。誰も上書き不可
2コマンド引数claude --...その起動だけの指定
3Project local.claude/settings.local.jsonこのリポの個人上書き（非コミット）
4Project shared.claude/settings.jsonチーム共有（コミット）
5User~/.claude/settings.json全プロジェクトの個人既定
マージ規則：配列は連結、オブジェクトはディープマージ、スカラーは上位が上書き。例外は fallbackModel（最上位が連鎖全体を供給）。例：user で許可・project で拒否 → project の拒否が勝つ。
03

permissions（最重要）

PERMISSIONS

日常の体験と安全を一番左右する。deny → ask → allow の順で評価、最初に当たったルールが勝つ。deny は bypass でも勝つ。

{ "permissions": { "allow": ["Bash(npm:*)", "Bash(git:*)", "Edit(src/**)"], "ask": ["Bash(git push:*)"], "deny": ["Bash(rm:*)", "Read(./.env)", "Read(./.env.*)", "Read(./secrets/**)", "WebFetch"], "defaultMode": "default", "disableBypassPermissionsMode": "disable" # チームでは推奨 } }
ルール書式
Tool(指定)。完全一致／前方prefix:*／パターン*。例：Bash(npm run test:*)、Read(./.env)。MCPツールは mcp__server__tool（二重アンダースコア・括弧なし）。
秘密情報は deny
.env・secrets/**・認証情報を deny: Read(...) で遮断。認証情報コミット防止の第一歩。
Read/Edit のパス記法
//path=絶対、~/path=ホーム、/path=設定ファイル基準の相対、path・./path=カレント基準。
複合コマンドは自動分割
&&・||・;・パイプで繋いだ Bash は個別コマンドに分解して評価される。繋げば通る抜け道にはならない。
defaultMode / モード挙動
defaultルール外は都度確認（標準）
acceptEditsファイル編集は自動承認、シェル等は確認。プロトタイピング向き
plan変更せず計画のみ（Shift+Tab で入る Plan）
dontAskallow 済み以外は確認せず拒否。固定ツール面の headless 向き
bypassPermissions全スキップ＝yolo。隔離環境のみ。--dangerously-skip-permissions
チームの肝：誰か1人が bypass にすると全ルールが無力化する。Managed か共有設定で "disableBypassPermissionsMode": "disable" を入れておく。

05

CLAUDE.md（メモリ）

MEMORY

起動時に自動で読まれる常駐文脈。簡潔さが命。肥大は性能劣化に直結する。

読まれる階層
組織 → ~/.claude/CLAUDE.md（個人グローバル）→ リポ直下 CLAUDE.md（共有）→ 作業中のサブディレクトリの CLAUDE.md。下位が状況に応じて足される。
作る・足す・直す
/init で生成、「覚えて」と頼んでその場追記（auto memory）、/memory で編集。
@インポート
@README.md や @~/.claude/personal.md で他ファイルを取り込み（最大4階層）。肥大化を分割で防ぐ。
中身の指針
スタック・主要コマンド・規約・禁止事項を要点だけ。長い説明書にしない。効かない記述は棚卸で削除。
.claude/rules/ で分割
ルールを複数ファイルに分割管理。frontmatter の paths でファイル種別ごとに適用範囲を絞れる。
06

hooks（ゲート化）

HOOKS

フック無しだと CLAUDE.md の規約は“お願い”。フックで強制ゲートになる。AIゲート運用の実装層。

主なイベント発火タイミング・できること
PreToolUseツール実行前。allow / deny / ask / 入力改変。最強の防御点（本番ファイル保護・危険コマンド遮断）
PostToolUse実行後。整形・チェック・出力の書き換え（秘密のマスク等）
UserPromptSubmit送信時に文脈を注入
SessionStart / Endセッション開始・終了（直近コミットの注入など）
Stop / SubagentStop応答終了時。最終チェック・継続強制
PreCompact圧縮前の処理
// 例：書き込み後に Prettier を自動実行（承認連打を撲滅） { "hooks": { "PostToolUse": [{ "matcher": "Write|Edit", "hooks": [{ "type": "command", "command": "npx prettier --write \"$CLAUDE_TOOL_INPUT_FILE_PATH\"" }] }] } }
ハンドラ種別（5種）
command（シェル）／http（エンドポイント）／mcp_tool（MCPツール）／prompt（単発LLM判定）／agent（サブエージェント検証）。
matcher / 終了コード
ツール名の正規表現（Edit|Write、mcp__.*）。PreToolUse で exit 2 = 遮断。matcher 無しは毎回発火。既定タイムアウト10分（イベント別の上書きあり）。
governance：PreToolUse で「本番ファイル編集」「git push」を要レビューに、PostToolUse で品質チェック、Stop で最終検証。これで規約が文書からenforced gateになる。CI/CD とも連携可。
07

MCP 設定

MCP

外部ツール接続。スコープ＝どこまで共有するかを意識して追加する。

$ claude mcp add github --scope project -- npx -y @modelcontextprotocol/server-github $ claude mcp list # 一覧 $ claude mcp get github # 詳細 # セッション中は /mcp で接続・認証・状態を確認
スコープ保存先共有範囲
local（既定）~/.claude.json自分・このプロジェクトだけ
project.mcp.json（コミット）チーム全員で共有
user~/.claude.json自分の全プロジェクト
共有のコツ：チームで使うサーバは --scope project で .mcp.json に入れてコミット。各自は enableAllProjectMcpServers や /mcp で有効化。鍵は env や各自の認証で渡し、.mcp.json に秘密を直書きしない。
08

どの仕組みを使う？

WHICH

機能が多いので、まず「やりたいこと」から引く。同じことを CLAUDE.md に書くだけでは“守られない”点が肝。

こうしたい使う仕組み性質
毎回同じ前提・規約を効かせるCLAUDE.md知識（助言）
何を許可・拒否するか決めるsettings.json / permissions境界
規約を機械的に守らせる・自動化hooks強制
繰り返す手順・専門知識を教える（自動で効く）skills手順（自動発火）
よく使うプロンプトを手動で即呼ぶslash command手順（手動）
別文脈で並行・レビュー役を立てるsubagents（/agents）役割
外部ツール・データに繋ぐMCP接続
迷ったら：「助言なら CLAUDE.md、強制なら hook、手順なら skill」。CLAUDE.md に“〜禁止”と書いても破られるなら、hook で固める。
09

トラブルシュート

FIX

詰まったら、まず /status と /config。「いま効いてる層」を見れば大半は切り分く。

症状原因・対処
設定が効かない／反映されない優先順位。/status の Setting sources で効いてる層を確認。上位スコープが上書きしていないか。再起動も試す
許可したのに Permission denieddeny が先に勝つ。広い deny（例 Bash(npm:*)）が個別 allow を潰していないか
Model not available／起動しないmodel がアカウントの tier に無い。override を外すか、使えるモデルに変える
書いたのに丸ごと無視される~/.claude.json に書いていないか（別物・スキーマエラーで黙殺）。settings.json へ移す
フックが発火しないmatcher の正規表現（MCPは mcp__server__.* 必須）／exit code（2＝遮断）／既定10分タイムアウト／スクリプトの実行権限
壊した！設定はタイムスタンプ付きで自動バックアップされる（保持世代数は版により異なるため要確認）。直前の動く版に戻せる
チーム導入・最小セット
これをコミットすれば全員同じ土台
# リポに置いてコミット <repo>/CLAUDE.md # スタック・規約・禁止事項（簡潔に） <repo>/.mcp.json # 共有MCP（秘密は直書きしない） <repo>/.claude/settings.json # 下記を入れる └─ permissions.allow # 通す定番コマンド └─ permissions.deny # rm / .env / secrets / WebFetch └─ disableBypassPermissionsMode: "disable" └─ hooks.PostToolUse # 自動整形 └─ hooks.PreToolUse # 本番/ pushを要レビュー（ゲート） # 各自ローカル（gitignore） .claude/settings.local.json # 個人の上書きだけ

原則は「共有はコミット・個人は local・強制はフック・秘密は deny」。優先順位（Managed > 引数 > local > 共有 > user）と /status を覚えておけば事故らない。

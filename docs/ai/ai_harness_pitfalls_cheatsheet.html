ハーネスの落とし穴.

AIエージェント導入 ― 失敗あるある & 最小構成

ハーネスは、AIに安全に作業させるための仕組み。最初から作り込まず、まず最小限の安全から始める。

前半は“何が壊れるか”、後半は“最小で何を置くか”。実装の詳細は設定早見表、任せる判断は線引きシート、事故対応は振り返りシートへ。ここは束ねる役。

DON'T FALL
落ちない
AIが勝手に暴走・破壊しない。権限とサンドボックスで被害の上限を決める。
STOPPABLE
止められる
人が途中で止められる。不可逆な操作は承認制にする。
TRACEABLE
追える
何をしたか後で分かる。作業ログ・記録を必ず残す。
最小ハーネスの組み立て順
入口→ 権限→ 作業ログ→ レビュー観点→ 人間の判断→ rollback
PART 1

落とし穴（失敗あるある）

01

やりがちな7つ

PITFALLS

どれも「賢さ」でなく仕組みの不足で起きる。症状 → 対策で押さえる。

① プロンプトインジェクション → ツール実行
症状：外部入力に紛れた指示で、AIが“仕事の一部”として危険操作。
対策：入力元を信用しない・権限を絞る・実行を記録。
② Shadow AI（野良エージェント）
症状：各自が勝手に使い、誰が何に繋いでるか不明。事故っても追えない。
対策：禁止でなく、使ってよい入口・権限・対象を定義。
③ CI通過 ≠ 完了
症状：Lint・テスト通過＝リリースOKと錯覚。業務影響や計測は見ていない。
対策：「技術チェック / 実装判断 / 業務リスク受容」を分ける。
④ AIレビューが感想文
症状：「全体的に良いです」で返り、事故点（設定・連携・権限）を見ない。
対策：レビュー観点を固定（仕様/環境/外部連携/権限/ログ）。
⑤ 生成コードが別環境で動かない
症状：手元では動くが別環境で死ぬ。依存・OS・バージョン差分。
対策：差分レビューだけでなく、クリーン環境での再現確認。
⑥ CLAUDE.md が肥大化
症状：ルール・禁止・例外を足し続け、巨大な館内規則に。
対策：完全版を狙わず、入口→禁止→レビュー→証跡の順で育てる。
⑦ 周辺の仕組みが本体
症状：権限・文脈・ログ・hooks・MCP…と増え、もはや小さな開発基盤。
対策：「ツール選定」でなく「最小の基盤設計」として扱う。全部は最初からやらない。
02

参照軸：OWASP Agentic Top 10

REFERENCE

AIセキュリティを「プロンプト漏洩対策」で止めない。OWASP が Agentic Applications 向け Top 10（2026・2025年12月公開）を出している。主なリスク（抜粋）：

コードリスク
ASI01目標の乗っ取り（外部入力・ツール出力で目的を歪める）
ASI02ツールの悪用（過剰権限・曖昧な指示で正規ツールを誤用）
ASI03認証情報・権限の悪用（継承/キャッシュした権限の乱用）
ASI04供給網の汚染（ツール・モデル・ペルソナの改ざん）
ASI05想定外のコード実行
ASI06メモリ・文脈の汚染（RAG・記憶の毒入れ）
ASI07エージェント間通信の不備（多エージェント連携の悪用）
ASI08連鎖的な障害（1つの失敗が次々に波及）
ASI10暴走エージェント（意図から逸脱した挙動）
キモは Least Agency：自律は“既定”でなく“獲得するもの”。「何にアクセスできるか」だけでなく「確認なしでどこまで動けるか」を絞る。全10項目は OWASP 公式（genai.owasp.org）を参照。
系譜：このAgentic版は OWASP LLM Top 10（プロンプトインジェクション＝LLM01 等）の発展形。単発のLLMでなく、計画し・行動するエージェント特有のリスク（権限・ツール・多エージェント・自律）に軸足を移している。
PART 2

最小構成

03

まず満たす3つ

MINIMUM

最初に必要なのは全部ではなく、この3つだけ。

落ちない
AIが勝手に暴走・破壊しない。権限を絞り、サンドボックスで被害の上限を決める。
止められる
人が途中で止められる。送信・本番反映・削除など不可逆操作は承認制。
追える
何をしたか後で分かる。見た/変えたファイル・判断・未確認を記録。
最小ライン：この3つが満たせていれば、まず動かしてよい。残りは走りながら足す。
人のゲートはどこに置く？：操作ごと（最も厳）／計画ごと（plan を承認）／成果物ごと（PR・差分で承認）。影響が大きいほど手前で、小さく可逆なら後ろ（事後監査）。＝線引きシートの自律度①〜④の置き換え。
04

最小ハーネス（6つ）

TEMPLATE

この6つを埋めれば、最小限の安全は確保できる。コピーして使う。

# 最小ハーネス 01_入口 対象リポジトリ / やらせる範囲 / やらせない範囲 02_権限 読取 / 書込 / コマンド実行 / 外部API / secret接触は禁止 03_作業ログ 見たファイル / 変えたファイル / 判断したこと / 未確認のこと 04_レビュー 仕様整合 / 影響範囲 / 例外系 / セキュリティ / 計測 / rollback 05_人間判断 採用 / 差し戻し / 保留 / リスク受容 06_復旧 rollback手順 / 撤退ライン（どうなったら止めるか）
05

巨大化を止める

STOP BLOAT

ハーネス整備は、放っておくと「全部整えたい→本体が進まない」に陥る。毎回この一言で切る：「これは将来必要。でも今回のゲートではない。」

2つの問いだけ：① 今回の最小安全条件は何か？　② 今回やらないが台帳に残すものは何か？　CLAUDE.md 等も完全版を狙わず、Phase 0 入口 → 1 禁止 → 2 レビュー → 3 証跡、と段階で育てる。
06

アンチパターン

PITFALLS
最初から完璧を狙う
全部整えてから始める＝本体が進まない。まず最小限から。
セキュリティ＝漏洩対策だけ
権限・ツール実行・メモリ・外部入力まで見る。
権限を絞らない
必要最小限（Least Agency）。自律は獲得するもの。
ログを残さない
事故っても追えない。記録は最優先の1つ。
AIレビュー任せ
感想文を信用しない。観点を固定する。
CLAUDE.md 肥大
最初から完全にしない。段階で育てる。
まず、この3つ
全部を最初から、ではなく最小から。
AIが勝手に落ちないか（権限・上限）
人が途中で止められるか（承認）
何をしたか追えるか（ログ）
今回のゲートを絞ったか

結論：まず「落ちない・止められる・追える」を満たし、巨大化は“今回のゲートではない”で止める。整備は段階的に、本体を止めない範囲で。

つながり：実装の詳細 → 設定早見表（permissions / hooks / MCP）｜ 任せる判断 → 任せる線引き｜ 事故対応 → 振り返り・障害対応。

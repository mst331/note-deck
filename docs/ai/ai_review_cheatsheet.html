AI出力のレビュー・検収の型.

確かめてから採用する

流暢さは、正しさではない。目的適合・事実・安全を観点で確かめ、受け入れ基準で検収してから採用する。

「任せる線引き」の片割れ。任せた出力を“感想文レビュー”で通さず、観点で見て、独立した手段で裏取りし、要件の受け入れ条件で判定する。

DON'T TRUST FLUENCY
流暢さで信じない
なめらかで自信ありげでも、正しいとは限らない。口調と正確さは無関係。
VERIFY
裏取りで確かめる
出典・再現・別手段で検証。重要なほど厳しく。同じAIに聞くだけでは弱い。
ACCEPT BY CRITERIA
基準で検収
要件の受け入れ条件で判定。採用 / 修正 / 却下を人が決める。
レビュー・検収の流れ
受け取る→ 観点でレビュー→ 裏取り・再現→ 受け入れ基準で検収→ 採用 / 修正 / 却下
01

大前提：流暢さ ≠ 正しさ

WHY

AI出力は読みやすく、自信ありげ。だからこそ誤りが見えにくい。検証は“疑い”から始める。

もっともらしい誤り
事実・数値・出典を、自然な文章で“それっぽく”誤る（ハルシネーション）。
自信 ≠ 正しさ
断定口調でも間違える。口調の強さは正確さの根拠にならない。
文脈の取りこぼし
暗黙の前提・社内事情・最新状況を外す。指示にない所を勝手に補う。
部分最適
局所は正しいが、全体の目的・整合からズレることがある。
02

レビュー観点（共通）

CHECKLIST

毎回同じ観点で見る。観点を固定すると、見落としと“感想文化”を防げる。

# AI出力レビュー観点（共通） □ 目的適合 依頼の目的・要件に合っているか □ 事実・数値 固有名詞 / 数字 / 引用 / URL は裏取りしたか □ 論理 筋が通っているか・飛びはないか □ 完成度 抜け・例外・エッジケースの考慮はあるか □ 安全・権限 機密 / 権限 / セキュリティ上の問題はないか □ スタイル 読み手・形式・トーンに合っているか
03

出力タイプ別の観点

BY TYPE

出力の種類で、特に警戒する所が変わる。

出力特に見る所
文章事実誤り / 誇張・断定 / 構成の筋 / 読み手への適合

要約重要点の欠落 / 歪曲・誇張 / 原文にない追加
翻訳誤訳 / ニュアンス / 用語の一貫性
04

検証の方法

VERIFY

「読んで納得」は検証ではない。外部の事実・実行で確かめる。

出典で裏取り
一次情報・公式に当たる。AIが示した出典も、実在するか開いて確認。
自分で再現
コードは実行、計算は検算。動かして確かめる。
別手段でクロスチェック
検索・別AI・人。独立した手段で突き合わせる。
重要度で強度を変える
影響が大きい・不可逆なほど厳しく。低リスクは軽く。
落とし穴：同じAIに「合ってる?」と聞くと「合ってます」と返りがち。自己検証は補助であって保証ではない。最終確認は独立した手段で。
05

ハルシネーションの見分け

RED FLAGS

特に“捏造されやすい”要素を知っておくと、警戒すべき所が分かる。

固有名詞・数値・日付
もっともらしい捏造の定番。必ず裏取り。
引用・URL・出典
存在しないものを作ることがある。開いて実在確認。
API・関数・ライブラリ
存在しない/古いものを自信ありげに。ドキュメント確認・実行で検証。
知識の境界
最新・ニッチ・社内固有は外しやすい。境界付近ほど疑う。
06

検収（受け入れ基準）

ACCEPT

レビューの最後は判定。要件の受け入れ条件で、採用/修正/却下を決める。

# 検収（判定） 対象 {AI出力} 受け入れ条件 {要件のDone・Given-When-Then} 判定 採用 / 修正 / 却下 理由 {事実 / 論理 / 完成度 のどこか} 修正指示 {どこを・なぜ・どうしてほしいか}
修正指示は具体的に：「直して」でなく「どこが・なぜ・どうしてほしい」。受け入れ条件は要件定義の型の Done をそのまま使える。
07

レビューを仕組みにする

SYSTEM

毎回ブレないように、レビューを個人技から仕組みへ。

観点を固定する
02のチェックリストを毎回使う。AIにレビューさせる時も観点を渡す。
採用は人が承認
AIレビューは補助。最終的な「採用」の判断は人が持つ。
検証を記録する
何を確認し、何を直したかを残す。後で追える＝品質が積み上がる。
影響大は二重チェック
不可逆・高影響は、別の人/手段でもう一度。
08

アンチパターン

PITFALLS
流暢だから採用
読みやすさ＝正しさではない。中身で見る。
AIの自信を信用
断定でも裏取り。口調は根拠でない。
一部だけ見て全体承認
冒頭OKで通す。抜けは後半・例外系に潜む。
「いい感じです」レビュー
観点のない感想。固定観点で具体的に。
同じAIに検証させて安心
独立した手段で。自己検証は補助。
出典を確認せず引用
存在しないURL・論文も。開いて確かめる。
検収 1枚チェック
採用する前にこれを問う
流暢さでなく中身で見たか
事実・数値・出典を裏取りしたか
コードは動かして確かめたか
受け入れ条件で検収したか
独立した手段で検証したか
採用は人が判断したか

結論：流暢さは正しさではない。観点で見て、独立した手段で裏取りし、受け入れ条件で検収する。これが「任せる」の片割れ。

つながり：任せる判断 → 任せる線引き｜ 受け入れ条件 → 要件定義の型｜ 安全に動かす → ハーネスの落とし穴｜ 失敗の振り返り → 振り返り・障害対応。

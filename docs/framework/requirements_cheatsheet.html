要件定義の型.

業務 → 機能 → 非機能 を曖昧さなく固める

要件の曖昧さは、そのまま後工程の手戻りになる。先に「スコープ・優先度・完了条件」を決める。

フレームワーク（MoSCoW・ユーザーストーリー/INVEST・非機能分類・Given-When-Then）を1枚に。ベンダー依頼・社内実装どちらの起点にも。

BUSINESS
業務要件
なぜ / 誰の課題
解決したい課題・目的・効果。これが全ての土台。ここがブレると全部ブレる。
FUNCTIONAL
機能要件
何をするか
システムが提供する振る舞い。ユーザーストーリーと受け入れ条件で表す。
NON-FUNCTIONAL
非機能要件
どの品質で
性能・可用性・セキュリティ等。落とすと本番事故。数値で定義する。
要件定義の流れ
目的(なぜ)→ As-Is / To-Be→ スコープ→ 機能(ストーリー)→ 非機能→ 優先度(MoSCoW)→ 受け入れ条件
01

3層で捉える

LAYERS

要件は3層。上の層が下の層の理由になる。「なぜ」から降ろすのが原則。

層答える問い成果物の例
業務要件なぜやる？ 誰の何の課題を解く？目的・KGI/KPI・対象業務・効果
機能要件何ができればいい？ユーザーストーリー・画面・処理・データ
非機能要件どの品質で動けばいい？性能・可用性・セキュリティ等の数値目標
02

目的と背景を固める

WHY

最初に「なぜ」を言語化。ここが弱いと、機能の取捨選択ができなくなる。

5W1H で輪郭
Why(目的) / Who(対象) / What(課題) / When(時期) / Where(範囲) / How(手段の方向性)。
As-Is → To-Be
現状の問題（As-Is）と、ありたい姿（To-Be）。その差分が要件になる。
効果を測れる形に
「改善する」ではなく「離脱率を○%下げる」。KGI/KPIで成否を判定可能に。
コツ：「解決策」ではなく「課題」を書く。Reactで作るは手段。要件は○○が△秒でできることのように達成状態で書く。
03

スコープ定義（やる / やらない）

SCOPE

最大の防衛ライン。「やらないこと」を明記するのが、スコープクリープ対策。

項目書くこと
In（やる）今回の対象機能・対象範囲・対象ユーザー
Out（やらない）今回は対象外。誤解を防ぐため明示的に書く
前提成立の条件（既存API継続・データ移行済み 等）
制約予算・期限・技術・法令・体制の縛り
ベンダー依頼の肝：In/Out が曖昧だと見積りも責任分界も割れる。「○○は今回Out」と書くだけで、後の「言った言わない」を大幅に減らせる。
04

機能要件（ユーザーストーリー）

FUNCTIONAL

「誰が・何を・なぜ」で書くと、価値が消えない。受け入れ条件とセットで。

# ユーザーストーリー As a 〈役割 / ペルソナ〉 I want 〈機能・行動〉 so that 〈得たい価値・目的〉 # 受け入れ条件（Given-When-Then） Given 〈前提・状態〉 When 〈操作・イベント〉 Then 〈期待される結果〉
INVEST（良いストーリーの条件）：Independent 独立 / Negotiable 交渉余地 / Valuable 価値がある / Estimable 見積れる / Small 小さい / Testable テスト可能。大きすぎる物語は分割する。
05

非機能要件（品質の軸）

NON-FUNCTIONAL

忘れられがちで、落ちると本番事故になる。各軸に必ず数値目標を。

軸決めること（例）
性能・拡張性応答時間（例: 2秒以内）/ 同時接続 / ピーク / 将来の増加見込み
可用性稼働率（例: 99.9%）/ 障害時の復旧目標（RTO/RPO）/ メンテ窓
セキュリティ認証・権限 / 暗号化 / 脆弱性対応 / 個人情報・法令対応
運用・保守性監視・ログ / バックアップ / デプロイ / 障害対応フロー
移行性既存データ移行 / 旧URLのリダイレクト / 切替方式・切り戻し
互換・ユーザビリティ対応ブラウザ・端末 / アクセシビリティ / 多言語
禁句を数値に：「速い」→「2秒以内」、「大量」→「同時1万」、「適宜」→条件を明記。曖昧語のまま通すと、検収で必ず揉める。
06

優先度（MoSCoW）

PRIORITY

全部を同時にはできない。4段階で仕分け、Must を絞るほど計画が締まる。

区分意味判断
Must無いと成立しない（必須）満たせなければリリース不可
Should重要だが代替・延期が可能余裕があれば今回
Couldあると嬉しい工数が余れば
Won't今回はやらない（明示）次回以降。スコープ防衛
目安：全部 Must にしない。Must は工数の6割程度までに抑える（DSDM由来の目安）。残りに調整余地があると炎上しにくい。
07

完了条件 & トレーサビリティ

DONE

「終わり」を先に決める。要件は ID で追えるようにし、変更は記録する。

受け入れ条件＝完了の定義
各要件に「これが満たされたら完了」を Given-When-Then で。検収・テストの根拠になる。
要件ID で追跡
REQ-001 のようにIDを振り、設計・実装・テストに紐づける。抜け漏れが見える。
変更管理
途中の要件変更は「いつ・誰が・なぜ」を記録。スコープと優先度を再評価する。
08

アンチパターン

PITFALLS
曖昧語を放置
「速い・適宜・柔軟に」。数値・条件に落とす。
手段を要件にする
「Reactで」ではなく「○○できること」。手段は設計で決める。
全部 Must
優先度が無い＝調整不能。仕分けを必ず。
非機能の抜け
性能・可用性・移行・セキュリティの欠落は本番事故に直結。
受け入れ条件が無い
「終わり」が未定義＝検収で揉める。
Out を書かない
やらないことが曖昧＝スコープが膨張する。
09

記入例（ワークスルー）

EXAMPLE

1つの要件を最初から最後まで通すと、各フレームワークの繋がりが見える。題材: 主要LPの問い合わせフォーム改善。

# 題材: 主要LPの問い合わせフォーム改善 【目的(なぜ)】 問い合わせ完了率が低い(現状 12%)。離脱を減らしCVを増やす。 効果指標: フォーム完了率 12% → 18% 【As-Is → To-Be】 As-Is: 入力項目が多くスマホで離脱。エラーで最初からやり直し。 To-Be: 項目を最小化・入力途中保存・その場でエラー表示。 【スコープ】 In : 入力項目の見直し / インラインバリデーション / スマホ最適化 Out: 会員登録機能 / CRM連携の改修（今回は Won't） 前提: 既存の送信APIは継続利用 制約: 既存トンマナ踏襲 / 6月末まで 【機能要件（ユーザーストーリー）】 As a スマホで見ている見込み客 I want 必須項目だけで素早く送りたい so that 途中で諦めずに問い合わせできる 受け入れ条件: Given 必須項目が未入力のまま When 送信ボタンを押す Then 該当項目にその場でエラー表示、入力済みは保持 【非機能要件（抜粋）】 性能 : フォーム表示 2秒以内 / 送信応答 1秒以内 可用性 : 営業時間帯の稼働率 99.9% セキュリティ: 通信はTLS / 個人情報は保存せずAPI転送のみ 互換 : iOS・Android 最新2世代 / Safari・Chrome 【優先度（MoSCoW）】 Must : 必須項目の最小化 / インラインバリデーション Should : 入力途中の保存 Could : 住所自動補完 Won't : CRM連携の改修
10

要件定義書の目次テンプレ

TEMPLATE

そのまま“書く成果物”の雛形。コピーして各章を埋めれば要件定義書になる。

# 要件定義書（テンプレ） 0. 概要 / 改訂履歴 1. 目的・背景 なぜ・誰の課題・効果指標(KGI/KPI) 2. スコープ In / Out / 前提 / 制約 3. 機能要件 ユーザーストーリー + 受け入れ条件（要件IDを振る） 4. 非機能要件 性能 / 可用性 / セキュリティ / 運用 / 移行 / 互換（数値目標） 5. 優先度(MoSCoW) Must / Should / Could / Won't 6. 制約・前提・リスク 予算 / 期限 / 技術 / 法令 / 体制 7. 用語定義 関係者で言葉を揃える 8. 関連資料 / 参照 既存仕様・チケット・議事録
要件定義 1枚チェック
これが言えたら要件は固まっている
なぜやるか、効果は測れるか
In / Outが書いてあるか
機能は誰の何の価値か（ストーリー）
非機能に数値目標があるか
優先度をMoSCoWで仕分けたか
各要件に受け入れ条件があるか

結論：スコープのIn/Outを書く・優先度はMoSCoW・完了は受け入れ条件で・曖昧語は数値化・手段でなく課題を書く。この5つで、後工程の手戻りはぐっと減る。

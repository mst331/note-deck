振り返りの型.

振り返り・障害対応 ― KPT / ポストモーテム

振り返りは犯人探しでなく、仕組み直し。人を責めず、事実 → 原因 → 対策を回す。

軽い振り返りは KPT、障害の深掘りは ポストモーテム。タイムライン・なぜなぜ・恒久対策で、同じ失敗を繰り返さない仕組みにする。

BLAMELESS
人を責めない
問題にするのは人でなく仕組み・プロセス。責めると事実が隠れ、再発する。
KPT
軽い振り返り
Keep / Problem / Try。定例やプロジェクト末に。Try は具体的に1〜3個。
POSTMORTEM
障害の深掘り
タイムライン → 根本原因 → 恒久対策。担当と期限をつけて横展開。
障害対応の流れ
検知→ 止血(暫定復旧)→ タイムライン記録→ なぜなぜ(根本原因)→ 恒久対策→ 横展開・共有
01

まず原則（Blameless）

PRINCIPLE

最重要。人でなく仕組みを問題にする。責めると、次から事実が出てこなくなる。

仕組みを責める
「誰がミスした」でなく「なぜミスできてしまう仕組みだったか」。同じ状況なら誰でも起こす前提で。
心理的安全性
正直に話して不利にならない場をつくる。隠蔽が一番のリスク。
事実と解釈を分ける
起きたこと（事実）と、こうだろう（解釈）を混ぜない。空・雨・傘の作法。
02

KPT（軽い振り返り）

RETRO

定例やプロジェクト末の定番。短時間で「続ける・直す・試す」を出す。

# KPT Keep（続ける） うまくいったこと・続けたいこと Problem（課題） 困った・うまくいかなかったこと Try（次やる） 次に試すこと （具体的に1〜3個・担当をつける）
コツ：Try を出して終わりにしない。担当と期限をつけ、次回の Keep/Problem で結果を見る。回すから効く。
派生：Try に Action を明示した KPTA、日本発の YWT（やったこと／わかったこと／つぎやること）も同系統。チームに馴染む型を選べばよい。
03

障害の初動

RESPONSE

障害発生時は復旧（止血）が最優先。原因究明は後。並行して記録を始める。

止血を優先
まず影響を止める（ロールバック・切替・遮断）。原因の完全解明は復旧後でよい。
記録を即開始
何時に何をしたかをその場で残す。後で思い出せない。タイムラインの素になる。
役割を決める
指揮（判断）・対応（手を動かす）・連絡（社内外共有）を分ける。一人で抱えない。
状況共有
分かっている事実だけを定期共有。「不明」も正直に。憶測を断定で流さない。
重大度目安対応
Sev1主要機能の全停止・広範な影響即時・全力。経営にも即共有
Sev2一部機能の障害・回避策あり当日対応。関係者へ共有
Sev3軽微・限定的通常運用の中で対応
04

タイムライン

TIMELINE

時刻ベースで事実だけを時系列に。後の分析と文書の土台になる。

# タイムライン（事実のみ・時刻つき） 10:02 監視アラート発報（5xx 急増） ← 検知 10:15 原因切り分け開始 10:40 直近デプロイをロールバック ← 止血 10:52 エラー率が平常へ ← 復旧 11:30 根本原因の調査開始 # 解釈・推測はここに混ぜない（別欄に）
計測の観点：「発生→検知」までの時間（MTTD）と「検知→復旧」までの時間（MTTR）を出す。検知が遅いほど被害が伸びるので、まず検知を早める改善が効く。
05

根本原因分析（なぜなぜ）

ROOT CAUSE

表層で止めない。「なぜ」を繰り返し、仕組みの原因まで降りる。

# なぜなぜ分析（5 Whys） 事象: {何が起きたか} なぜ①? {直接の原因} なぜ②? {その原因} なぜ③? {さらに} なぜ④? {さらに} なぜ⑤? {根本原因 → ここに対策を打つ}
「人の不注意」で止めない
そこで止まったら「なぜ不注意が起こせる仕組みか」を更に問う。仕組みが根本。
原因は1つとは限らない
線形に詰めすぎない。特性要因図（フィッシュボーン：人/方法/設備/環境…）で複数要因も洗う。
06

恒久対策

FIX

暫定（その場しのぎ）と恒久（再発防止）を分ける。対策は3方向で考える。

起きにくく
発生確率を下げる。仕組み・自動化・ガード（例：誤操作できないUI、CIで検知）。
早く気づく
検知を速くする（MTTD短縮）。監視・アラート・しきい値。気づくのが遅い障害ほど痛い。
影響を小さく
起きても被害を限定（段階リリース・切り戻し容易化・冗長化）。
対策にオーナーと期限
「気をつける」は対策ではない。チケット化して担当・期限・完了条件を付ける。
07

ポストモーテム文書

DOC

Blameless で書き、共有して横展開する。他システムにも同じリスクが無いか確認するのが肝。

# ポストモーテム（Blameless） 1. 概要 何が・いつ・どれくらいの時間 2. 影響 ユーザー / 売上 / 範囲・継続時間 3. タイムライン 時刻ベースの事実（検知→止血→復旧） 4. 根本原因 なぜなぜ / 特性要因（表層でなく根本） 5. 対策 暫定 / 恒久（担当・期限つき） 6. 学び・横展開 他にも同じリスクは? 共有先は?
運用：重大度の高い障害（Sev1/2）は必ず実施。対策は完了まで追跡し、後日のレビューで閉じる。書いて終わり・対策やりっぱなしが最大の無駄。
08

アンチパターン

PITFALLS
犯人探し
人を責めると事実が隠れ、再発する。仕組みを責める。
「気をつけます」で終了
精神論は対策でない。仕組み・自動化に落とす。
表層原因で止める
直接原因だけ直して再発。なぜを繰り返す。
対策が放置
オーナー・期限なしのTODOは消える。チケット化。
振り返り自体をやらない
忙しさで飛ばす。短くても必ず回す。
学びを共有しない
同じ穴に別チームが落ちる。横展開する。
振り返り 1枚チェック
これが言えたら次に活きる
人でなく仕組みを責めたか
事実と解釈を分けたか（時刻つき）
なぜを繰り返し根本原因まで降りたか
対策は起きにくく/早く気づく/影響小か
対策に担当・期限があるか
学びを横展開したか

結論：犯人探しでなく仕組み直し。事実→根本原因→（担当つき）恒久対策を回し、横展開する。これで同じ失敗を繰り返さない。

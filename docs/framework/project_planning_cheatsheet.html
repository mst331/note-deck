計画の型.

プロジェクト計画 ― 分解・見積り・段取り・リスク

計画は「当てる」ためでなく、ズレを早く見つけるためにある。だから分解・見積り・責任・リスクを一巡させる。

要件定義の次の工程。WBS・3点見積り・クリティカルパス・RACI・リスク管理を1枚に。完璧な計画より、更新し続けられる計画を。

DECOMPOSE
分解（WBS）
成果物ベースで作業に割る。見えていない作業は見積れない。まず分解。
ESTIMATE
見積り
点でなく幅で。3点見積りと過去実績で、不確実性を見える化する。
DE-RISK
リスク管理
起きてから対応しない。確率×影響で評価し、先回りで備える。
計画の流れ
スコープ→ WBS(分解)→ 見積り→ 依存・順序→ スケジュール→ RACI→ リスク
01

計画に含めるもの

OVERVIEW

計画書の骨子はこの8つ。コピーして埋めれば計画のたたき台になる。

# プロジェクト計画（骨子） 1. 目的・成功基準 何が達成できたら成功か 2. スコープ In / Out（要件定義から引き継ぐ） 3. WBS / 成果物 作業分解（ワークパッケージ） 4. 見積り 規模(S/M/L) / 期間（3点見積り） 5. スケジュール マイルストーン / 依存 / クリティカルパス 6. 体制(RACI) R / A(1人) / C / I 7. リスク 発生確率 × 影響 / 対応策 / トリガー 8. 前提・制約 予算 / 期限 / 体制 / 外部依存
02

WBS（作業分解）

DECOMPOSE

大きな塊は見積れない。成果物ベースで、扱える粒度まで割る。

成果物で分ける
「動詞の作業」より「できあがる物」で分割すると抜けにくい。
100%ルール
子の合計＝親の100%。足して全体になり、重複もない（＝MECE）。
粒度の目安
末端のワークパッケージは数日〜2週間。大きすぎると進捗が見えない。
抜け漏れ点検
設計・テスト・レビュー・リリース・ドキュメント・移行も作業。忘れがち。
03

見積り（規模・期間）

ESTIMATE

見積りは点でなく幅。当たらない前提で、ばらつきを見える化する。

# 3点見積り（PERT） 期待値 = (楽観 O ＋ 4×最可能 M ＋ 悲観 P) / 6 ばらつき σ = (P − O) / 6 # 例: O=2, M=4, P=10 日 期待値 = (2 + 16 + 10) / 6 = 4.7 日 σ = (10 − 2) / 6 = 1.3 → 4.7 ± 1.3 日
相対見積り（S/M/L）
絶対時間より相対サイズが速くブレにくい。ストーリーポイント／プランニングポーカー。
過去実績で補正
似た作業の実測（ベロシティ）を基準に。自分の勘より過去データ。
計画錯誤に注意：人は自分の作業を楽観的に見積もる（カーネマン）。幅で持つ・他者にレビューさせる・過去実績で殴る。
04

依存・順序・クリティカルパス

SEQUENCE

作業には順序がある。全体の納期を決めるのは最長経路＝クリティカルパス。

依存を引く
先行→後続を明示。並行できる作業は並行に。前提待ちを早く出す。
クリティカルパス（CPM）
遅れると全体が遅れる最長の経路。ここを重点監視。短縮はここから。
フロート（余裕）
CP以外の作業が持つ遅延吸収幅。ここは多少遅れても全体に響かない。
外部依存は早出し
ベンダー納品・他部署・APIなど、自分で握れない依存ほど前倒しで確認。
05

スケジュール & マイルストーン

SCHEDULE

日付の羅列でなく判断ポイントを置く。バッファは散らさず集約する。

マイルストーン
「進む/直す/止める」を判断する節目。成果物の完成や検収に置く。
バッファは末尾に集約
各タスクに余裕を散らすと食い潰される（学生症候群・パーキンソンの法則）。末尾にまとめる＝クリティカルチェーン(CCPM)の考え方。
ガントは結果
WBS＋依存＋見積りができれば線は引ける。線から作らない。
計画は更新する
マイルストーンごとに実績で引き直す。固定した計画は嘘になる。
ローリングウェーブ
直近は細かく、先は粗く計画する。不確実な将来を作り込みすぎない。長期・多サイトの移行向き。
06

体制・役割（RACI）

ROLES

「誰がやる/誰が決める」を作業ごとに固定する。Accountable は必ず1人。

記号意味ルール
R Responsible実行する人複数可
A Accountable説明責任・最終承認必ず1人
C Consulted相談する（双方向）必要な人だけ
I Informed報告を受ける（一方向）絞る
定番の事故：A が複数＝決まらない、A が不在＝止まる。各作業に A をちょうど1人。ベンダー連携では責任分界点もここで握る。
07

リスク管理

DE-RISK

洗い出し → 発生確率 × 影響で評価 → 高い順に対応策 → トリガーを決めて見直す。

対応中身例
回避リスク源を断つ（やり方を変える）不確実な新技術をやめる
軽減確率/影響を下げるPoC・段階リリース・冗長化
転嫁第三者に移す外注・契約・保険
受容受け入れて備えるコンティンジェンシー（予備）
リスク登録簿：リスク／確率／影響／対応／トリガー（何が起きたら発動）／担当 を一覧に。定例で見直す。「起きてから対応」は計画ではない。
08

アンチパターン

PITFALLS
楽観見積り
計画錯誤。幅で持つ・過去実績・他者レビュー。
バッファを散らす
各タスクの余裕は食い潰される。末尾に集約。
A が複数 / 不在
決まらない・止まる。各作業に1人。
依存を無視
並行できない作業を並行前提に。CPを引く。
リスク未管理
「起きてから」。登録簿で先回り。
計画を更新しない
計画は生もの。節目で実績で引き直す。
09

記入例（ワークスルー）

EXAMPLE

1つのプロジェクトを通すと各要素の繋がりが見える。題材: 1サイトの Next.js 移行（縮約）。

# 例: 1サイトの Next.js 移行（縮約） 【成功基準】 旧と同等の表示・計測を維持し、主要URLの順位を落とさない 【WBS（抜粋）】 1. 棚卸・URL対応表 2. 実装（テンプレ移植/フォーム） 3. リダイレクト設計(301/308) 4. 計測タグ移行 5. テスト・検収 6. 切替・公開 【見積り（3点・日）】 実装: O=4 M=6 P=12 → 期待値 6.7 ± 1.3 日 【クリティカルパス】 URL対応表 → 実装 → リダイレクト → 検収 → 切替 （対応表が遅れると全部遅れる ＝ 最重点） 【RACI（抜粋）】 実装: R=担当 / A=自分 / C=ベンダー / I=関係部署 切替: R=インフラ / A=自分 / C=ベンダー / I=営業 【上位リスク】 ・対応表の漏れ → 軽減: 二重チェック / クロール突合 ・切替後のループ → 軽減: ステージングで全URL検証 ・計測タグ欠落 → 受容+検知: 公開直後にダッシュ確認
計画 1枚チェック
これが言えたら計画は回る
成功基準が定義されているか
WBSに抜けはないか（100%）
見積りを幅で持っているか
クリティカルパスを把握しているか
各作業に A が1人か
上位リスクに対応策があるか

結論：計画は当てるためでなく、ズレを早く見つけるため。分解・見積り(幅)・責任(A=1人)・リスクを一巡し、マイルストーンで更新し続ける。

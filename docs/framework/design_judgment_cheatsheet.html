設計判断の型.

PART 2 ― 人が握る側

失敗の多くは能力不足でなく、支配軸（その判断で一番重視すべき軸）の取り違え。まず「これは何の判断か」を名指しする。

レビュー・AIへの指示・判断ログで「なんか嫌」を「これは"◯◯"だ」に変えるための共通語彙（札）。どれも禁止ではなく「いつなら許されるか（敢えて選ぶ条件）」がある。A〜C群は人の設計判断、D〜E群はAI時代に急増し、検収・ハーネスシートと地続き。

16パターンの名称・概念: kawasima「Decision Quality と設計判断失敗パターン」を参考に、設計レビュー・AI活用向けに再構成。A〜E群の分類・合言葉は独自整理。

A — REUSE FIRST
既存を先に見る
標準・既存部品・既存規約・責務を見てから書く。再発明とバラつきを防ぐ。
B — RIGHT-SIZE
身の丈で作る
要件と運用体制に釣り合う規模で作る。将来の拡張は来てから。
C — DON'T SHRINK
縮こまらない
組織を低く見て弱い選択を続けない。互換は"誰が使っているか"で判断。
D — DON'T SKIP ★AI
確認を飛ばさない
「通った」を「できた」にしない。完了の定義を先に固定する。
E — BUILD IN ★危険
後付けできないものは最初から入れる
権限・履歴・監査・冪等性は後付け不可。依存は実在確認。
A

REUSE FIRST — 既存を見ていない／バラバラになる

標準・既存部品・規約・責務を確認してから書く。書き始める前の1分が、重複と乖離を防ぐ。

名前一言例合言葉
車輪の再発明 標準で足りるのに自作 標準バリデーションがあるのに各画面で独自実装 それ標準で足りない？
隣を見ない再実装 既存の共通部品を読まず似たものを新規作成 日付整形ユーティリティがあるのに formatDate/toDate が増殖 同じもの、もう無い？
郷に従わぬ正論 既存規約を無視して"正しさ"を新規分だけ持ち込む 既存はService層トランザクションなのに新規だけController処理 周りのコードに合わせた？
越境実装 担当の責務を越えて書く 上限判定をドメイン層でなく画面側(View)に書く それ、ここで判定する話？
許容条件：既存が明確に劣化している／規約自体を更新する合意がある時は、新規側で正しさを入れてよい。
B

RIGHT-SIZE — 作りすぎ／規模・将来を読み違える

要件と運用体制に釣り合う規模で作る。変化点は観測されてから、拡張は担当と時期が決まってから。

名前一言例合言葉
牛刀をもって鶏を割く 小さい要件に大げさな構成 1ページのLP改修に汎用基盤を持ち出す 規模に釣り合ってる？
早すぎる抽象化 変化点が見える前に汎用化 1種類しかないのに汎用フォームレンダラ 2回以上同じの出た？
取らぬ狸の拡張点 "将来やるかも"だけで拡張点を作る 「将来SMSも送るかも」で汎用通知基盤 時期と担当、決まってる？
身の丈に合わない設計 技術は正しいが運用できる人がいない 運用体制が無いのに重いインフラ／回せない量の運用ルール うちで運用できる？
許容条件：変化点が2回以上実際に出た／規模・運用体制が追いついている時だけ。
C

DON'T SHRINK — 縮こまりすぎ／成長機会を潰す

組織のスキルを実態より低く見積もらない。互換の実利用者を確認せず最大限保守的に振る舞わない。

名前一言例合言葉
羹に懲りて膾を吹く 組織を低く見て弱い選択を続ける 「保守できる人いない」でずっと旧いまま "保守できない"が言い訳になってない？
過去への忖度 互換の実利用者を確認せず最大限保守的に 使われていない旧仕様のため新旧並走で複雑化 それ、誰がまだ使ってる？
許容条件：実利用者・移行コストを確認した上で、なお守る価値があると判断できた時。
D

DON'T SKIP — 確認を飛ばす

「通った」を「できた」にしない。完了条件を入力・状態・期待結果に割ってから着手する。

名前一言例合言葉
早合点 "通った"を"できた"と勘違い 送信が200で「成功」、実は連携先で未受理／バッチ完走で対象0件 コマンドが通っただけでは？
見切り発車 仕様・受入基準が曖昧なまま実装 業務ルール未定なのにバリデーションだけ先に作る 完了の定義、決まってる？
名ばかりの完了 完了条件が判定不能 「正しく表示」「現行と同等」だけ＝測れない 入力・状態・期待結果に割れてる？
場当たり対応 短期で動かすため整合を壊す エラーを握りつぶして画面だけ進める／テスト期待値を実装に寄せる 一次対応？なら戻し条件と恒久チケット残した？
詳細はリンク先（再説明しない）：早合点・名ばかりの完了 → AI出力の検収（受け入れ基準・流暢さ≠正しさ）｜早合点 → ハーネスの落とし穴（CI通過≠完了）｜場当たり対応 → 振り返りの型（恒久対策）・ハーネス（rollback／撤退ライン）。
E

BUILD IN — 最初から入れるべきものを抜く

権限・履歴・監査・冪等性・状態遷移は後付け不可。依存は一次資料で実在確認してから使う。

名前一言例合言葉
制約の後付け 権限・履歴・監査・冪等性・状態遷移を後回し CRUDを作ってから権限／更新を作ってから監査ログ。フォーム・連携・決済・個人情報で致命的 制約は後付け不可。最初から入った？
砂上の依存 存在しないライブラリ・関数・引数を使う AIが実在しないパッケージ/引数を生成（slopsquatting） そのAPI、本当に在る？一次資料で確認した？
詳細はリンク先（再説明しない）：制約の後付け → ハーネスの落とし穴（権限・ログを最初から）｜砂上の依存 → AI出力の検収（存在しないAPIのレッドフラグ）・ハーネス（OWASP ASI04 供給網汚染）。
つながり：任せる前の整理 → 要件定義の型（見切り発車の受け入れ条件） ｜ 確認 → AI出力の検収 ｜ 安全に動かす → ハーネスの落とし穴 ｜ 事故後 → 振り返りの型

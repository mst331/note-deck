Claudeの作法.

Claude / Claude Code ベストプラクティス

コンテキストは埋まるほど性能が落ちる。多くの作法は、この一点から来ている。

だから「計画してから」「文脈は軽く」「難度に合わせる」が効く。以下は公式ドキュメントの芯を1枚に圧縮したもの（出典は末尾）。

PLAN FIRST
計画してから実装
まず Plan モード。計画を読んでから動かす。本番コードを“勢い”で書かない。
KEEP CONTEXT LIGHT
文脈は軽く保つ
こまめに圧縮、切り替えはリセット。文脈の劣化が最大の失敗モード。
SIMPLE > COMPLEX
単純が強い
凝った多エージェントより単純な制御ループ。役割は小さく、人がレビュー席に。
高品質な1サイクル
探索→ 計画 (Plan)→ 計画をレビュー→ 実装→ テストで検証→ /clear
01

プロンプトの原則

PROMPTING

公式の現行リファレンス（Prompting best practices）の技法を実務で効く形に整理。AI は“文脈ゼロの優秀な新人”だと思って書く。

1
明確・具体・直接的に
曖昧さを消す。何を・どの形で欲しいかを正確に。やってほしいことを言う（「〜するな」より「〜して」）。
2
文脈と「なぜ」を渡す
目的・制約・背景を添える。理由が分かると狙い通りに動きやすい。
3
例を見せる（multishot）
形式や概念は例で示すのが一番速い。最新モデルは例の細部までよく見る。
4
考えさせる
難しい推論は思考させてから答えさせる。今は思考モード（現行世代は adaptive thinking）が手動の「順を追って」より基本的に有効。仕様は世代で変わるので現行は公式で確認。
5
XML タグで構造化
Claude は <instructions> <context> <examples> のような区切りに強い。複雑な指示で効く。
6
役割を与える / 形式を指定
立場（system）で口調と範囲を決める。出力形式（JSON/Markdown等）も明示。役割は盛りすぎない。
7
長文は指示を後ろ / 複雑は分割
長い入力は指示を末尾に置くと拾われやすい。大きな課題はプロンプトチェーンで小分けに。
02

まず計画（Plan → 実装）

PLAN

どの一次情報も「実装前の計画」を非交渉の前提に挙げている。

Plan モードで入る
Shift + Tab で Plan に。まず計画を出させ、人が読んでから実行に移す。いきなり書かせない。
探索 → 計画 → 実装 → コミット
先にコードを探索・把握させ、計画を立て、それから書く。順番が品質を決める。
計画は“残す”
書いた計画・設計判断はファイルに残す（living plan / dev docs）。後続セッションの文脈になる。
本番は vibe coding しない
使い捨てなら勢いでも可。本番コードは構造化した思考・検証・ドキュメントが要る。
03

モデル × 思考量

MODEL

難度にコストを合わせる。難しいほど“賢いモデル＋思考多め”、単純で大量なら“軽いモデル＋思考少なめ”。

タスクの性質目安モデル思考量
難しい設計・原因不明のデバッグ・アーキ判断Opus多め（拡張思考オン）
日常の実装・リファクタ・レビューSonnet標準
大量・定型・分類・要約などの軽作業Haiku少なめ
考え方：思考モードは「効くが遅くて高い」。難しい一手にだけ厚く振り、定型では薄くする。/model で切替、思考量は /effort で上下。最上位ティアは Fable 5（思考は常時オン）。現行のモデル構成は公式で確認。
04

セッションの重さ・移動

CONTEXT

文脈の劣化が最大の失敗モード。重さを“見て・減らして・切り替える”を習慣に。

見る残量を確認
/context で使用量を把握。埋まる前に動く。
減らすこまめに圧縮
長い作業は限界前に /compact。残す要点を引数で指定も可。
切るタスクが変わったら
/clear で新規。引きずらず作り直す方が結局速い。
分けるレビューは別セッション
新しい文脈は自分の書いたコードに肩入れしない。Writer / Reviewer を分ける。
回す大量作業は並列
移行・一括処理は claude -p をファイル単位でループ。権限は --allowedTools で絞る。
05

エージェント / スキル設計

BUILD

「単純 > 複雑」が大原則。賢く作るより、小さく・確実に・監査できる形に。

まず単純な制御ループ
多エージェントの前に、低レベルツール（Bash/Read/Edit）＋必要な抽象だけ。凝った構成は最後の手段。
スキルは description が命
発火するかは name+description で決まる。「何を・いつ使うか」を三人称で具体的に。最新世代は強い言葉（MUST等）だと過発火するため断定調は避ける。
progressive disclosure
本体 SKILL.md は小さく。詳細は別ファイルに分け、必要時だけ読ませる。多数入れても文脈を食わない設計。
1スキル1役 / 手順は番号で
複雑な処理は番号付きの手順・チェックリストに。役割を絞ると壊れにくい。
MCPツールは完全修飾名
GitHub:create_issue のようにサーバ接頭辞付きで。曖昧だと取り違える。
信頼できる元のみ・監査
スキルは任意コードを実行し得る。自作 or Anthropic 由来を基本に、導入前に全ファイルを確認。
06

メモリ管理・棚卸（CLAUDE.md）

MEMORY

CLAUDE.md は毎回読み込まれる“常駐コスト”。肥大は劣化に直結する。

簡潔に保つ
スタック・コマンド・規約・“やってはいけない罠”を要点だけ。長文の説明書にしない。
その場で足す / 編集する
気づきは「覚えて」と頼んで追記（auto memory）、整えるときは /memory で編集。
プロジェクト用と個人用を分ける
リポ直下＝チーム共有、~/.claude＝自分用。共有物に個人設定を混ぜない。
定期棚卸＝削除ファースト
増やすより消す。効いていない記述・古い規約は落とす。“追加して直らないなら構造を疑う”。

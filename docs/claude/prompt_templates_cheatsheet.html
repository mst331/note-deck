プロンプトの型.

実務テンプレート集 ― 流し込むだけ

いいプロンプト＝明確・文脈・例・形式指定。型を持てば、毎回ゼロから書かなくていい。

基本骨格に、レビュー・バグ調査・移行・ドキュメント・技術選定・文書作成の型を用意。{…} を埋めて使う。タグ名は自由（日本語でも可）だが、ここでは標準的な英語タグで統一。

BE EXPLICIT
明確・具体に
曖昧さを消す。何を・どの形で欲しいかを正確に。「〜するな」より「〜して」。
GIVE CONTEXT
文脈と「なぜ」
目的・背景・制約を渡す。理由が分かると狙い通りに動く。
SHOW THE SHAPE
例・形式を示す
出力形式を指定し、例を1つ。形が固まると品質が安定する。
01

基本骨格（これに流し込む）

SKELETON

迷ったらこの6ブロック。以下の各テンプレも全部この変形。

<role>{あなたは“どんな専門家”か}</role> <context> {プロジェクト・背景}。目的は {なぜこれをやるか}。 </context> <instructions> {やってほしいこと。具体的に・手順で} 確証がない箇所は推測と明記。不明点は実装前に質問。 </instructions> <input> {対象のコード／ログ／文章} </input> <output_format> {箇条書き／表／JSON／PREP など。例を1つ} </output_format> <constraints> {守る条件・やってはいけないこと} </constraints>
タグは自由：<入力> など日本語でもよい。大事なのは「区切りで役割を分ける」こと。難しいタスクは Plan モードや思考オン、ダメなら分割（プロンプトチェーン）。
02

コードレビュー

REVIEW

PR・差分の品質チェック。重要度つきで指摘がほしい時。

<role>厳しめだが建設的なコードレビュアー。</role> <context>Next.js(App Router)。本番前のPR。重視: 可読性・パフォーマンス・セキュリティ。</context> <instructions> 下の差分をレビューし、指摘を重要度(高/中/低)で分類。 各指摘に「理由」と「修正案(コード)」を付ける。良い点も1〜2個。 確証がない所は推測と明記。 </instructions> <input> {git diff をここに貼る} </input> <output_format> 表: 重要度 / ファイル:行 / 指摘 / 理由 / 修正案 </output_format>
03

バグ調査・原因究明

DEBUG

ログ・エラーから原因の当たりをつけたい時。事実と推測を分ける。

<role>原因究明が得意なSRE。</role> <context>環境: {EC2 / Next.js / CloudFront 等}。症状: {何がいつ起きているか}。</context> <instructions> 下のログ/エラーから、考えられる原因を可能性の高い順に挙げる。 各原因に「根拠(ログのどの行か)」「確認方法(コマンド)」「対処」を付ける。 事実と推測を分けて書く。情報が足りなければ何が必要か言う。 </instructions> <input> {ログ / エラー / 再現手順} </input> <output_format> 原因(高→低) / 根拠 / 確認コマンド / 対処 </output_format>
04

移行・リファクタ

MIGRATE

PHP/Blade/WP → Next.js の移植。対応関係も一緒に出させる。

<role>PHP→Next.js移行に詳しいエンジニア。</role> <context>元: {Laravel Blade / WP テンプレ}。先: Next.js App Router(v16, RSC)。</context> <instructions> 下を Server Component に移植する。 - ルーティング / データ取得 / メタデータ の対応表を示す - ブラウザ動作が要る所だけ 'use client' - 変更理由をコメントで。不明点は実装前に質問 </instructions> <input> {元のコード} </input> <output_format> 移植後コード + 対応表(旧→新) + 注意点 </output_format>
05

ドキュメント化・要約

DOC

コード・議事録・設計判断を、読み手に合わせて要約。

<role>テクニカルライター。</role> <context>読み手: {新メンバー / 経営層}。用途: {オンボーディング / 報告}。</context> <instructions> 下を {300字 / 1ページ} で要約。結論を先に(PREP)。 専門用語は最小限。事実と解釈を分ける。 </instructions> <input> {コード / 議事録 / 設計判断} </input> <output_format> 結論 → 理由 → 具体例 → 結論。見出し付き。 </output_format>
06

設計・技術選定（比較）

DECIDE

選択肢を中立に比較し、前提つきで推奨がほしい時。

<role>中立な技術アドバイザー。立場を持ちすぎない。</role> <instructions> {課題} に対する選択肢を2〜3個挙げ、比較表で評価。 評価軸: {コスト / 学習コスト / 保守性 / 移行容易性}。 各案のメリット・デメリット・向くケースを書く。 最後に前提つきで推奨を1つ。反対意見も添える。 </instructions> <output_format> 比較表 + 推奨(前提つき) + 主なリスク </output_format>
07

文書作成（PREP / エスカレーション）

WRITE

上長・経営向けの報告やエスカレーション。事実と解釈を分けて。

<role>社内向け報告・エスカレーション文書の起草者。</role> <context>宛先: {上長 / 経営}。経緯: {時系列の事実}。</context> <instructions> PREP(結論→理由→具体例→結論)で起草。 「事実」と「私の解釈」を明確に分ける。 感情語・過度な断定を避ける。引用は発言者と日時を明記。 依頼事項を最後に箇条書き。 </instructions> <input> {時系列ログ / チャット抜粋} </input> <output_format> 結論 / 事実(時系列) / 解釈 / 依頼事項 </output_format>
08

要件整理 → チケット化

TICKET

ざっくり要望を、Backlog 等の実装可能なチケットに分解したい時。

<role>要件を実装可能な単位に分解するPM／テックリード。</role> <context>課題管理: {Backlog 等。プロジェクト / チケット形式}。読み手: 実装担当。</context> <instructions> 下のざっくり要望を、実装可能なチケットに分解する。 各チケットに「目的」「受け入れ条件(Done)」「想定作業」「依存」「規模(S/M/L)」を付ける。 曖昧な点は {確認事項} として別出しにする。 </instructions> <input> {要望・背景} </input> <output_format> チケット一覧(タイトル / 目的 / Done / 作業 / 依存 / 規模) ＋ 確認事項 </output_format>
09

テスト生成

TEST

関数・コンポーネントのテストを、観点を網羅して書かせたい時。

<role>テスト設計が得意なエンジニア。</role> <context>対象: {言語/FW 例: TypeScript + Vitest}。方針: 正常系＋異常系＋境界値。</context> <instructions> 下の関数/コンポーネントのテストを書く。 - 正常系・異常系・境界値を網羅し、各テストの意図をコメント - モックが要る依存は明示 - 仕様が不足する所は {要確認} として挙げる </instructions> <input> {対象コード} </input> <output_format> テストコード ＋ カバーした観点の一覧 ＋ 要確認 </output_format>
10

効かせるコツ

TIPS
明確・具体に
曖昧を消す。複数解釈できる指示は割れる。
「なぜ」を書く
目的・背景があると狙い通りに動く。
例を1つ見せる
形式・粒度が一発で固まる。
出力形式を指定
表 / JSON / PREP など。後工程が楽。
長文は指示を後ろに
大量の入力の後に指示を置くと拾われやすい。
難しいタスクは思考オン
Plan モード／思考モード（adaptive thinking）。難所にだけ厚く。
ダメなら分割
1プロンプトで欲張らず、段階に分ける。
やることを言う
「〜するな」より「〜して」。肯定形で。
骨格だけ覚える
これさえあれば毎回作れる
<role> どんな専門家か <context> 背景 ＋ なぜ <instructions> やること（具体・手順） <input> 対象（コード/ログ/文章） <output_format> 形式（表/JSON/PREP）＋例1つ <constraints> 守る条件・禁止 # プラスα ・例を1つ見せる ・出力形式を指定 ・長文は指示を後ろ ・難所は思考オン ・ダメなら分割

結論：役割・文脈(なぜ)・具体指示・入力・出力形式・制約の6点を埋めるだけ。あとは例を1つと形式指定で、品質はだいたい安定する。
